Honest Status

Actually working now

  • The Next.js dashboard is built and deployed on Vercel.
  • The GitHub repository is connected and deploys to production.
  • The page shows the social content workflow end to end.
  • Protected API routes exist for seeding and publishing.
  • The Vercel Cron route exists and is scheduled once per day.
  • The dashboard can now read Firestore data when Firebase is configured and seeded.

Not live yet

  • No real LinkedIn, Instagram, TikTok, YouTube, Threads, Bluesky, or X account is connected inside the app yet.
  • The visible post data is demo data unless Firestore has been seeded.
  • The dashboard buttons mostly update the browser view; full write-back to Firestore is a next step.
  • Real posting requires Buffer credentials and real Buffer channel IDs.
  • Direct native platform APIs are not implemented in this MVP.

How Page Data Loads

The homepage is a server-rendered Next.js page. It asks lib/dashboard-data.ts for dashboard data before rendering. 

app/page.tsx
  -> getDashboardData()
    -> if Firebase Admin env vars exist:
         read campaigns, channels, posts from Firestore
       else:
         use lib/sample-data.ts
Why this matters: before this change, the dashboard always used sample data. Now it can show real Firestore records after the Firebase service account and seed step are complete. If Firebase is missing or empty, it intentionally falls back to demo data so the page still loads.

API Routes

POST /api/admin/seed

Copies the demo campaigns, channels, and posts into Firestore. Protected by x-admin-secret.

If Firebase Admin is missing, it returns review-mode information and does not write to Firestore.

POST /api/posts/publish

Sends one post to Buffer when BUFFER_API_KEY and a Buffer channel ID exist. Protected by x-admin-secret.

If Buffer is missing, it returns a mock response instead of pretending a real post was sent.

GET /api/cron/publish

Vercel calls this daily. It looks for due scheduled posts in Firestore and attempts to send them to Buffer.

Protected by CRON_SECRET. Without Firebase or Buffer, it reports review mode and sends nothing.

Environment Variables

Public Firebase browser values

NEXT_PUBLIC_FIREBASE_API_KEY
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN
NEXT_PUBLIC_FIREBASE_PROJECT_ID
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID
NEXT_PUBLIC_FIREBASE_APP_ID

These identify the Firebase web app. They are not the service account key.

Private server values

FIREBASE_PROJECT_ID
FIREBASE_CLIENT_EMAIL
FIREBASE_PRIVATE_KEY
ADMIN_API_SECRET
CRON_SECRET
BUFFER_API_KEY
BUFFER_DEFAULT_CHANNEL_ID

These must only live in Vercel environment variables or local .env.local. Never commit or paste the private key.

Data Model

Collection What it stores Example fields
campaigns Content themes and business goals. name, goal, owner, stage, startDate, endDate
channels Social accounts and publishing destinations. platform, handle, cadence, status, bufferChannelId
posts Drafts, approvals, scheduled posts, and publishing status. caption, status, scheduledAt, targetPlatforms, bufferPostId
publishLogs Audit trail for publishing attempts. postId, provider, externalId, status, createdAt

Post Status Flow

idea -> draft -> needs_approval -> approved -> scheduled -> publishing -> published

scheduled -> failed

The dashboard currently demonstrates this movement in the browser. The next engineering improvement is to make each button write the new status back to Firestore so the state is shared by the whole team.

Deployment Wiring

  1. Code lives in GitHub: milansevic22/crumblabz-social-engine.
  2. Vercel is connected to the GitHub repo.
  3. Every push to main can create a production deployment.
  4. Vercel installs dependencies with pnpm install --ignore-scripts.
  5. Vercel builds with pnpm build.
  6. Production serves from https://crumblabz-social-engine.vercel.app.
  7. Vercel Cron calls /api/cron/publish once daily.

Real Publishing Path

Firestore post with status scheduled
  -> Vercel Cron calls /api/cron/publish
    -> route checks CRON_SECRET
    -> route finds due posts in Firestore
    -> route calls createBufferPost()
    -> Buffer queues or schedules the social post
    -> Firestore post receives bufferPostId/status update
    -> publishLogs receives an audit record
Current limitation: the path is coded, but it only becomes real after Buffer is connected and each destination has a real Buffer channel ID. TikTok, Instagram, LinkedIn, and YouTube direct APIs are not the first version; Buffer is the first gateway.
